Questions about how to use GLFW should be asked either in the \href{https://discourse.glfw.org/c/support}{\texttt{ support section}} of the forum, under the \href{https://stackoverflow.com/questions/tagged/glfw}{\texttt{ Stack Overflow tag}} or \href{https://gamedev.stackexchange.com/questions/tagged/glfw}{\texttt{ Game Development tag}} on Stack Exchange or in the IRC channel {\ttfamily \#glfw} on \href{https://libera.chat/}{\texttt{ Libera.\+Chat}}.

Questions about the design or implementation of GLFW or about future plans should be asked in the \href{https://discourse.glfw.org/c/dev}{\texttt{ dev section}} of the forum or in the IRC channel. Please don\textquotesingle{}t open a Git\+Hub issue to discuss design questions without first checking with a maintainer.\hypertarget{md__c_1_2_users_2_fahad_2_one_drive_01-_01_seneca_2_seneca_2_sem__5_2_s_e_d500-_introduction_01t9c61a440ff9c26f30250e8beda307dd2_autotoc_md11}{}\doxysection{\texorpdfstring{Reporting a bug}{Reporting a bug}}\label{md__c_1_2_users_2_fahad_2_one_drive_01-_01_seneca_2_seneca_2_sem__5_2_s_e_d500-_introduction_01t9c61a440ff9c26f30250e8beda307dd2_autotoc_md11}
If GLFW is behaving unexpectedly at run-\/time, start by setting an \href{https://www.glfw.org/docs/latest/intro_guide.html\#error_handling}{\texttt{ error callback}}. GLFW will often tell you the cause of an error via this callback. If it doesn\textquotesingle{}t, that might be a separate bug.

If GLFW is crashing or triggering asserts, make sure that all your object handles and other pointers are valid.

For bugs where it makes sense, a short, self contained example is absolutely invaluable. Just put it inline in the body text. Note that if the bug is reproducible with one of the test programs that come with GLFW, just mention that instead.

{\bfseries{Don\textquotesingle{}t worry about adding too much information}}. Unimportant information can be abbreviated or removed later, but missing information can stall bug fixing, especially when your schedule doesn\textquotesingle{}t align with that of the maintainer.

{\bfseries{Please provide text as text, not as images}}. This includes code, error messages and any other text. Text in images cannot be found by other users searching for the same problem and may have to be re-\/typed by maintainers when debugging.

{\bfseries{Note\+:}} You must have all necessary \href{https://en.wikipedia.org/wiki/Intellectual_property}{\texttt{ intellectual property rights}} to any code you contribute. If you did not write the code yourself, you must explain where it came from and under what license you received it. Even code using the same license as GLFW may not be copied without attribution.

{\bfseries{There is no preferred patch size}}. A one character fix is just as welcome as a thousand line one, if that is the appropriate size for the fix.

In addition to the code, a complete bug fix includes\+:


\begin{DoxyItemize}
\item Change log entry in {\ttfamily README.\+md}, describing the incorrect behavior
\item Credits entries for all authors of the bug fix
\end{DoxyItemize}

Bug fixes will not be rejected because they don\textquotesingle{}t include all the above parts, but please keep in mind that maintainer time is finite and that there are many other bugs and features to work on.

If the patch fixes a bug introduced after the last release, it should not get a change log entry.

{\bfseries{Note\+:}} You must have all necessary rights to any code you contribute. If you did not write the code yourself, you must explain where it came from and under what license. Even code using the same license as GLFW may not be copied without attribution.

{\bfseries{Note\+:}} If you haven\textquotesingle{}t already implemented the feature, check first if there already is an open issue for it and if it\textquotesingle{}s already being developed in an \href{https://github.com/glfw/glfw/branches/all}{\texttt{ experimental branch}}.

{\bfseries{There is no preferred patch size}}. A one-\/character change is just as welcome as one adding a thousand lines, if that is the appropriate size for the feature.

In addition to the code, a complete feature includes\+:


\begin{DoxyItemize}
\item Change log entry in {\ttfamily README.\+md}, listing all new symbols
\item News page entry, briefly describing the feature
\item Guide documentation, with minimal examples, in the relevant guide
\item Reference documentation, with all applicable tags
\item Cross-\/references and mentions in appropriate places
\item Credits entries for all authors of the feature
\end{DoxyItemize}

If the feature requires platform-\/specific code, at minimum stubs must be added for the new platform function to all supported and experimental platforms.

If it adds a new callback, support for it must be added to {\ttfamily tests/event.\+c}.

If it adds a new monitor property, support for it must be added to {\ttfamily tests/monitor.\+c}.

If it adds a new Open\+GL, Open\+GL ES or Vulkan option or extension, support for it must be added to {\ttfamily \doxylink{glfwinfo_8c}{tests/glfwinfo.\+c}} and the behavior of the library when the extension is missing documented in {\ttfamily \doxylink{compat_8dox}{docs/compat.\+dox}}.

If you haven\textquotesingle{}t already, read the excellent article \href{https://chris.beams.io/posts/git-commit/}{\texttt{ How to Write a Git Commit Message}}.

Features will not be rejected because they don\textquotesingle{}t include all the above parts, but please keep in mind that maintainer time is finite and that there are many other features and bugs to work on.

Please also keep in mind that any part of the public API that has been included in a release cannot be changed until the next {\itshape major} version. Features can be added and existing parts can sometimes be overloaded (in the general sense of doing more things, not in the C++ sense), but code written to the API of one minor release should both compile and run on subsequent minor releases.
