The \doxylink{group__window_ga3c96d80d363e67d13a41b5d1821f3242}{GLFWwindow} object encapsulates both a window and a context. They are created with \doxylink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window} and destroyed with \doxylink{group__window_ga806747476b7247d292be3711c323ea10}{glfw\+Destroy\+Window}, or \doxylink{group__init_gafd90e6fd4819ea9e22e5e739519a6504}{glfw\+Terminate}, if any remain. As the window and context are inseparably linked, the object pointer is used as both a context and window handle.

To see the event stream provided to the various window related callbacks, run the {\ttfamily events} test program.\hypertarget{window_guide_window_creation}{}\doxysubsection{\texorpdfstring{Window creation}{Window creation}}\label{window_guide_window_creation}
A window and its Open\+GL or Open\+GL ES context are created with \doxylink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window}, which returns a handle to the created window object. For example, this creates a 640 by 480 windowed mode window\+:

The window handle is passed to all window related functions and is provided to along with all input events, so event handlers can tell which window received the event.\hypertarget{window_guide_window_full_screen}{}\doxysubsubsection{\texorpdfstring{Full screen windows}{Full screen windows}}\label{window_guide_window_full_screen}
To create a full screen window, you need to specify which monitor the window should use. In most cases, the user\textquotesingle{}s primary monitor is a good choice. For more information about retrieving monitors, see \doxysectlink{monitor_guide_monitor_monitors}{Retrieving monitors}{2}.

Full screen windows cover the entire display area of a monitor, have no border or decorations.

Windowed mode windows can be made full screen by setting a monitor with \doxylink{group__window_ga12fabf78575e59c00f822f323ae0b6ae}{glfw\+Set\+Window\+Monitor}, and full screen ones can be made windowed by unsetting it with the same function.

Each field of the \doxylink{struct_g_l_f_wvidmode}{GLFWvidmode} structure corresponds to a function parameter or window hint and combine to form the {\itshape desired video mode} for that window. The supported video mode most closely matching the desired video mode will be set for the chosen monitor as long as the window has input focus. For more information about retrieving video modes, see \doxysectlink{monitor_guide_monitor_modes}{Video modes}{2}.

Once you have a full screen window, you can change its resolution, refresh rate and monitor with \doxylink{group__window_ga12fabf78575e59c00f822f323ae0b6ae}{glfw\+Set\+Window\+Monitor}. If you only need change its resolution you can also call \doxylink{group__window_gae54d1f4915ded15e267ddd3f41496cd2}{glfw\+Set\+Window\+Size}. In all cases, the new video mode will be selected the same way as the video mode chosen by \doxylink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window}. If the window has an Open\+GL or Open\+GL ES context, it will be unaffected.

By default, the original video mode of the monitor will be restored and the window iconified if it loses input focus, to allow the user to switch back to the desktop. This behavior can be disabled with the \doxylink{window_guide_GLFW_AUTO_ICONIFY_hint}{GLFW\+\_\+\+AUTO\+\_\+\+ICONIFY} window hint, for example if you wish to simultaneously cover multiple monitors with full screen windows.

If a monitor is disconnected, all windows that are full screen on that monitor will be switched to windowed mode. See \doxysectlink{monitor_guide_monitor_event}{Monitor configuration changes}{2} for more information.\hypertarget{window_guide_window_windowed_full_screen}{}\doxysubsubsection{\texorpdfstring{"{}\+Windowed full screen"{} windows}{"Windowed full screen" windows}}\label{window_guide_window_windowed_full_screen}
If the closest match for the desired video mode is the current one, the video mode will not be changed, making window creation faster and application switching much smoother. This is sometimes called {\itshape windowed full screen} or {\itshape borderless full screen} window and counts as a full screen window. To create such a window, request the current video mode.

Window destruction always succeeds. Before the actual destruction, all callbacks are removed so no further events will be delivered for the window. All windows remaining when \doxylink{group__init_gafd90e6fd4819ea9e22e5e739519a6504}{glfw\+Terminate} is called are destroyed as well.

When a full screen window is destroyed, the original video mode of its monitor is restored, but the gamma ramp is left untouched.\hypertarget{window_guide_window_hints}{}\doxysubsection{\texorpdfstring{Window creation hints}{Window creation hints}}\label{window_guide_window_hints}
There are a number of hints that can be set before the creation of a window and context. Some affect the window itself, others affect the framebuffer or context. These hints are set to their default values each time the library is initialized with \doxylink{group__init_gab41771f0215a2e0afb4cf1cf98082d40}{glfw\+Init}. Integer value hints can be set individually with \doxylink{group__window_ga69c40728499720bef8a49aa925ea0efa}{glfw\+Window\+Hint} and string value hints with \doxylink{group__window_ga3c2270562d7ba13d92680ba1090adafe}{glfw\+Window\+Hint\+String}. You can reset all at once to their defaults with \doxylink{group__window_ga8050ddceed9dc6bd9d3aa35666195cd4}{glfw\+Default\+Window\+Hints}.

Some hints are platform specific. These are always valid to set on any platform but they will only affect their specific platform. Other platforms will ignore them. Setting these hints requires no platform specific headers or calls.

\begin{DoxyNote}{Note}
Window hints need to be set before the creation of the window and context you wish to have the specified attributes. They function as additional arguments to \doxylink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window}.
\end{DoxyNote}
\hypertarget{window_guide_window_hints_hard}{}\doxysubsubsection{\texorpdfstring{Hard and soft constraints}{Hard and soft constraints}}\label{window_guide_window_hints_hard}
Some window hints are hard constraints. These must match the available capabilities {\itshape exactly} for window and context creation to succeed. Hints that are not hard constraints are matched as closely as possible, but the resulting context and framebuffer may differ from what these hints requested.

{\bfseries{GLFW\+\_\+\+CLIENT\+\_\+\+API}} specifies which client API to create the context for. Possible values are {\ttfamily GLFW\+\_\+\+OPENGL\+\_\+\+API}, {\ttfamily GLFW\+\_\+\+OPENGL\+\_\+\+ES\+\_\+\+API} and {\ttfamily GLFW\+\_\+\+NO\+\_\+\+API}. This is a hard constraint.

\label{window_guide_GLFW_CONTEXT_CREATION_API_hint}%
\Hypertarget{window_guide_GLFW_CONTEXT_CREATION_API_hint}%
{\bfseries{GLFW\+\_\+\+CONTEXT\+\_\+\+CREATION\+\_\+\+API}} specifies which context creation API to use to create the context. Possible values are {\ttfamily GLFW\+\_\+\+NATIVE\+\_\+\+CONTEXT\+\_\+\+API}, {\ttfamily GLFW\+\_\+\+EGL\+\_\+\+CONTEXT\+\_\+\+API} and {\ttfamily GLFW\+\_\+\+OSMESA\+\_\+\+CONTEXT\+\_\+\+API}. This is a hard constraint. If no client API is requested, this hint is ignored.

An \doxysectlink{context_guide_context_glext_auto}{extension loader library}{2} that assumes it knows which API was used to create the current context may fail if you change this hint. This can be resolved by having it load functions via \doxylink{group__context_ga0e8af175218929615c16e74938c10f2a}{glfw\+Get\+Proc\+Address}.

\begin{DoxyNote}{Note}
@wayland The EGL API {\itshape is} the native context creation API, so this hint will have no effect.

@x11 On some Linux systems, creating contexts via both the native and EGL APIs in a single process will cause the application to segfault. Stick to one API or the other on Linux for now.

{\bfseries{OSMesa\+:}} As its name implies, an Open\+GL context created with OSMesa does not update the window contents when its buffers are swapped. Use Open\+GL functions or the OSMesa native access functions \doxylink{osmesa__context_8c_abdce3d5716b1a4b53266c171c5de4c8c}{glfw\+Get\+OSMesa\+Color\+Buffer} and \doxylink{osmesa__context_8c_a70e7b999151097a79a82572d7a70cb02}{glfw\+Get\+OSMesa\+Depth\+Buffer} to retrieve the framebuffer contents.
\end{DoxyNote}
\label{window_guide_GLFW_CONTEXT_VERSION_MAJOR_hint}%
\Hypertarget{window_guide_GLFW_CONTEXT_VERSION_MAJOR_hint}%
\label{window_guide_GLFW_CONTEXT_VERSION_MINOR_hint}%
\Hypertarget{window_guide_GLFW_CONTEXT_VERSION_MINOR_hint}%
{\bfseries{GLFW\+\_\+\+CONTEXT\+\_\+\+VERSION\+\_\+\+MAJOR}} and {\bfseries{GLFW\+\_\+\+CONTEXT\+\_\+\+VERSION\+\_\+\+MINOR}} specify the client API version that the created context must be compatible with. The exact behavior of these hints depend on the requested client API.

While there is no way to ask the driver for a context of the highest supported version, GLFW will attempt to provide this when you ask for a version 1.\+0 context, which is the default for these hints.

Do not confuse these hints with \doxylink{group__init_ga6337d9ea43b22fc529b2bba066b4a576}{GLFW\+\_\+\+VERSION\+\_\+\+MAJOR} and \doxylink{group__init_gaf80d40f0aea7088ff337606e9c48f7a3}{GLFW\+\_\+\+VERSION\+\_\+\+MINOR}, which provide the API version of the GLFW header.

\begin{DoxyNote}{Note}
{\bfseries{Open\+GL\+:}} These hints are not hard constraints, but creation will fail if the Open\+GL version of the created context is less than the one requested. It is therefore perfectly safe to use the default of version 1.\+0 for legacy code and you will still get backwards-\/compatible contexts of version 3.\+0 and above when available.

{\bfseries{Open\+GL ES\+:}} These hints are not hard constraints, but creation will fail if the Open\+GL ES version of the created context is less than the one requested. Additionally, Open\+GL ES 1.\+x cannot be returned if 2.\+0 or later was requested, and vice versa. This is because Open\+GL ES 3.\+x is backward compatible with 2.\+0, but Open\+GL ES 2.\+0 is not backward compatible with 1.\+x.

@macos The OS only supports forward-\/compatible core profile contexts for Open\+GL versions 3.\+2 and later. Before creating an Open\+GL context of version 3.\+2 or later you must set the \doxylink{window_guide_GLFW_OPENGL_FORWARD_COMPAT_hint}{GLFW\+\_\+\+OPENGL\+\_\+\+FORWARD\+\_\+\+COMPAT} and \doxylink{window_guide_GLFW_OPENGL_PROFILE_hint}{GLFW\+\_\+\+OPENGL\+\_\+\+PROFILE} hints accordingly. Open\+GL 3.\+0 and 3.\+1 contexts are not supported at all on mac\+OS.
\end{DoxyNote}
\label{window_guide_GLFW_OPENGL_FORWARD_COMPAT_hint}%
\Hypertarget{window_guide_GLFW_OPENGL_FORWARD_COMPAT_hint}%
{\bfseries{GLFW\+\_\+\+OPENGL\+\_\+\+FORWARD\+\_\+\+COMPAT}} specifies whether the Open\+GL context should be forward-\/compatible, i.\+e. one where all functionality deprecated in the requested version of Open\+GL is removed. This must only be used if the requested Open\+GL version is 3.\+0 or above. If Open\+GL ES is requested, this hint is ignored.

Forward-\/compatibility is described in detail in the \href{https://www.opengl.org/registry/}{\texttt{ Open\+GL Reference Manual}}.

Debug contexts for Open\+GL and Open\+GL ES are described in detail by the \href{https://www.khronos.org/registry/OpenGL/extensions/KHR/KHR_debug.txt}{\texttt{ GL\+\_\+\+KHR\+\_\+debug}} extension.

{\bfseries{GLFW\+\_\+\+OPENGL\+\_\+\+PROFILE}} specifies which Open\+GL profile to create the context for. Possible values are one of {\ttfamily GLFW\+\_\+\+OPENGL\+\_\+\+CORE\+\_\+\+PROFILE} or {\ttfamily GLFW\+\_\+\+OPENGL\+\_\+\+COMPAT\+\_\+\+PROFILE}, or {\ttfamily GLFW\+\_\+\+OPENGL\+\_\+\+ANY\+\_\+\+PROFILE} to not request a specific profile. If requesting an Open\+GL version below 3.\+2, {\ttfamily GLFW\+\_\+\+OPENGL\+\_\+\+ANY\+\_\+\+PROFILE} must be used. If Open\+GL ES is requested, this hint is ignored.

Open\+GL profiles are described in detail in the \href{https://www.opengl.org/registry/}{\texttt{ Open\+GL Reference Manual}}.

{\bfseries{GLFW\+\_\+\+CONTEXT\+\_\+\+ROBUSTNESS}} specifies the robustness strategy to be used by the context. This can be one of {\ttfamily GLFW\+\_\+\+NO\+\_\+\+RESET\+\_\+\+NOTIFICATION} or {\ttfamily GLFW\+\_\+\+LOSE\+\_\+\+CONTEXT\+\_\+\+ON\+\_\+\+RESET}, or {\ttfamily GLFW\+\_\+\+NO\+\_\+\+ROBUSTNESS} to not request a robustness strategy.

\label{window_guide_GLFW_CONTEXT_RELEASE_BEHAVIOR_hint}%
\Hypertarget{window_guide_GLFW_CONTEXT_RELEASE_BEHAVIOR_hint}%
{\bfseries{GLFW\+\_\+\+CONTEXT\+\_\+\+RELEASE\+\_\+\+BEHAVIOR}} specifies the release behavior to be used by the context. Possible values are one of {\ttfamily GLFW\+\_\+\+ANY\+\_\+\+RELEASE\+\_\+\+BEHAVIOR}, {\ttfamily GLFW\+\_\+\+RELEASE\+\_\+\+BEHAVIOR\+\_\+\+FLUSH} or {\ttfamily GLFW\+\_\+\+RELEASE\+\_\+\+BEHAVIOR\+\_\+\+NONE}. If the behavior is {\ttfamily GLFW\+\_\+\+ANY\+\_\+\+RELEASE\+\_\+\+BEHAVIOR}, the default behavior of the context creation API will be used. If the behavior is {\ttfamily GLFW\+\_\+\+RELEASE\+\_\+\+BEHAVIOR\+\_\+\+FLUSH}, the pipeline will be flushed whenever the context is released from being the current one. If the behavior is {\ttfamily GLFW\+\_\+\+RELEASE\+\_\+\+BEHAVIOR\+\_\+\+NONE}, the pipeline will not be flushed on release.

Context release behaviors are described in detail by the \href{https://www.opengl.org/registry/specs/KHR/context_flush_control.txt}{\texttt{ GL\+\_\+\+KHR\+\_\+context\+\_\+flush\+\_\+control}} extension.

{\bfseries{GLFW\+\_\+\+CONTEXT\+\_\+\+NO\+\_\+\+ERROR}} indicates whether errors are generated by the context. Possible values are {\ttfamily GLFW\+\_\+\+TRUE} and {\ttfamily GLFW\+\_\+\+FALSE}. If enabled, situations that would have generated errors instead cause undefined behavior.

{\bfseries{GLFW\+\_\+\+CONTEXT\+\_\+\+ROBUSTNESS}} indicates the robustness strategy used by the context. This is {\ttfamily GLFW\+\_\+\+LOSE\+\_\+\+CONTEXT\+\_\+\+ON\+\_\+\+RESET} or {\ttfamily GLFW\+\_\+\+NO\+\_\+\+RESET\+\_\+\+NOTIFICATION} if the window\textquotesingle{}s context supports robustness, or {\ttfamily GLFW\+\_\+\+NO\+\_\+\+ROBUSTNESS} otherwise.\hypertarget{window_guide_window_attribs_fb}{}\doxysubsubsection{\texorpdfstring{Framebuffer related attributes}{Framebuffer related attributes}}\label{window_guide_window_attribs_fb}
GLFW does not expose attributes of the default framebuffer (i.\+e. the framebuffer attached to the window) as these can be queried directly with either Open\+GL, Open\+GL ES or Vulkan.

If you are using version 3.\+0 or later of Open\+GL or Open\+GL ES, the {\ttfamily gl\+Get\+Framebuffer\+Attachment\+Parameteriv} function can be used to retrieve the number of bits for the red, green, blue, alpha, depth and stencil buffer channels. Otherwise, the {\ttfamily gl\+Get\+Integerv} function can be used.

The number of MSAA samples are always retrieved with {\ttfamily gl\+Get\+Integerv}. For contexts supporting framebuffer objects, the number of samples of the currently bound framebuffer is returned.

When calling {\ttfamily gl\+Get\+Framebuffer\+Attachment\+Parameteriv}, the red, green, blue and alpha sizes are queried from the {\ttfamily GL\+\_\+\+BACK\+\_\+\+LEFT}, while the depth and stencil sizes are queried from the {\ttfamily GL\+\_\+\+DEPTH} and {\ttfamily GL\+\_\+\+STENCIL} attachments, respectively.\hypertarget{window_guide_buffer_swap}{}\doxysection{\texorpdfstring{Buffer swapping}{Buffer swapping}}\label{window_guide_buffer_swap}
GLFW windows are by default double buffered. That means that you have two rendering buffers; a front buffer and a back buffer. The front buffer is the one being displayed and the back buffer the one you render to.

When the entire frame has been rendered, it is time to swap the back and the front buffers in order to display what has been rendered and begin rendering a new frame. This is done with \doxylink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfw\+Swap\+Buffers}.


\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfwSwapBuffers}}(window);}

\end{DoxyCode}


Sometimes it can be useful to select when the buffer swap will occur. With the function \doxylink{group__context_ga12a595c06947cec4967c6e1f14210a8a}{glfw\+Swap\+Interval} it is possible to select the minimum number of monitor refreshes the driver should wait from the time \doxylink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfw\+Swap\+Buffers} was called before swapping the buffers\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{group__context_ga12a595c06947cec4967c6e1f14210a8a}{glfwSwapInterval}}(1);}

\end{DoxyCode}


If the interval is zero, the swap will take place immediately when \doxylink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfw\+Swap\+Buffers} is called without waiting for a refresh. Otherwise at least interval retraces will pass between each buffer swap. Using a swap interval of zero can be useful for benchmarking purposes, when it is not desirable to measure the time it takes to wait for the vertical retrace. However, a swap interval of one lets you avoid tearing.

Note that this may not work on all machines, as some drivers have user-\/controlled settings that override any swap interval the application requests.

A context that supports either the {\ttfamily WGL\+\_\+\+EXT\+\_\+swap\+\_\+control\+\_\+tear} or the {\ttfamily GLX\+\_\+\+EXT\+\_\+swap\+\_\+control\+\_\+tear} extension also accepts {\itshape negative} swap intervals, which allows the driver to swap immediately even if a frame arrives a little bit late. This trades the risk of visible tears for greater framerate stability. You can check for these extensions with \doxylink{group__context_ga9a28c712d35f9e43534e1d03b051c04c}{glfw\+Extension\+Supported}.
