\hypertarget{news_news_33}{}\doxysection{\texorpdfstring{Release notes for version 3.\+3}{Release notes for version 3.3}}\label{news_news_33}
These are the release notes for version 3.\+3. For a more detailed view including all fixed bugs see the \href{https://www.glfw.org/changelog.html}{\texttt{ version history}}.

Please review the caveats, deprecations and removals if your project was written against an earlier version of GLFW 3.\hypertarget{news_features_33}{}\doxysubsection{\texorpdfstring{New features in version 3.\+3}{New features in version 3.3}}\label{news_features_33}
\hypertarget{news_gamepad_33}{}\doxysubsubsection{\texorpdfstring{Gamepad input via SDL\+\_\+\+Game\+Controller\+DB}{Gamepad input via SDL\_GameControllerDB}}\label{news_gamepad_33}
GLFW can now remap game controllers to a standard Xbox-\/like layout using a built-\/in copy of SDL\+\_\+\+Game\+Controller\+DB. Call \doxylink{group__input_gac9720b6e5062a5fc5fc0bb4be6dc04a0}{glfw\+Joystick\+Is\+Gamepad} to check if a joystick has a mapping, \doxylink{group__input_ga6e138d3b0761546f5a12546f367501d6}{glfw\+Get\+Gamepad\+State} to retrieve its input state, \doxylink{group__input_ga276a2aec236a671a99f6f0a9be7fa3fc}{glfw\+Update\+Gamepad\+Mappings} to add newer mappings and \doxylink{group__input_gab7a03c6ca8aa55a52b750d862d106ffe}{glfw\+Get\+Gamepad\+Name} and \doxylink{group__input_ga2c6a8b6e27cb8a8be7a0d020d9fd051c}{glfw\+Get\+Joystick\+GUID} for mapping related information.

For more information see \doxysectlink{input_guide_gamepad}{Gamepad input}{2}.\hypertarget{news_moltenvk_33}{}\doxysubsubsection{\texorpdfstring{Support for Vulkan on mac\+OS via Molten\+VK}{Support for Vulkan on macOS via MoltenVK}}\label{news_moltenvk_33}
GLFW now supports \href{https://moltengl.com/moltenvk/}{\texttt{ Molten\+VK}}, a Vulkan implementation on top of the Metal API, and its {\ttfamily VK\+\_\+\+MVK\+\_\+macos\+\_\+surface} window surface creation extension. Molten\+VK is included in the \href{https://vulkan.lunarg.com/}{\texttt{ mac\+OS Vulkan SDK}}.

For more information see \doxysectlink{vulkan_guide}{Vulkan guide}{0}.\hypertarget{news_content_scale_33}{}\doxysubsubsection{\texorpdfstring{Content scale queries for DPI-\/aware rendering}{Content scale queries for DPI-aware rendering}}\label{news_content_scale_33}
GLFW now provides content scales for windows and monitors, i.\+e. the ratio between their current DPI and the platform\textquotesingle{}s default DPI, with \doxylink{group__window_ga72718c983a9fffaa27f786bfe7d1c9ee}{glfw\+Get\+Window\+Content\+Scale} and \doxylink{group__monitor_ga28d856d94bea2ce8f09a67bb0346aaff}{glfw\+Get\+Monitor\+Content\+Scale}.

Changes of the content scale of a window can be received with the window content scale callback, set with \doxylink{group__window_gac6edcddbe785445d0d2b7b7be500836e}{glfw\+Set\+Window\+Content\+Scale\+Callback}.

The \doxylink{window_guide_GLFW_SCALE_TO_MONITOR}{GLFW\+\_\+\+SCALE\+\_\+\+TO\+\_\+\+MONITOR} window hint enables automatic resizing of a window by the content scale of the monitor it is placed, on platforms like Windows where this is necessary. This takes effect both on creation and when the window is moved between monitors. It is related to but different from \doxylink{window_guide_GLFW_COCOA_RETINA_FRAMEBUFFER_hint}{GLFW\+\_\+\+COCOA\+\_\+\+RETINA\+\_\+\+FRAMEBUFFER}.

For more information see \doxysectlink{window_guide_window_scale}{Window content scale}{2}.\hypertarget{news_setwindowattrib_33}{}\doxysubsubsection{\texorpdfstring{Support for updating window attributes}{Support for updating window attributes}}\label{news_setwindowattrib_33}
GLFW now supports changing the \doxylink{window_guide_GLFW_DECORATED_attrib}{GLFW\+\_\+\+DECORATED}, \doxylink{window_guide_GLFW_RESIZABLE_attrib}{GLFW\+\_\+\+RESIZABLE}, \doxylink{window_guide_GLFW_FLOATING_attrib}{GLFW\+\_\+\+FLOATING}, \doxylink{window_guide_GLFW_AUTO_ICONIFY_attrib}{GLFW\+\_\+\+AUTO\+\_\+\+ICONIFY} and \doxylink{window_guide_GLFW_FOCUS_ON_SHOW_attrib}{GLFW\+\_\+\+FOCUS\+\_\+\+ON\+\_\+\+SHOW} attributes for existing windows with \doxylink{group__window_gad39f43f07856efcf3d656426a9284c75}{glfw\+Set\+Window\+Attrib}.

For more information see \doxysectlink{window_guide_window_attribs}{Window attributes}{2}.\hypertarget{news_raw_motion_33}{}\doxysubsubsection{\texorpdfstring{Support for raw mouse motion}{Support for raw mouse motion}}\label{news_raw_motion_33}
GLFW now supports raw (unscaled and unaccelerated) mouse motion in disabled cursor mode with the \doxylink{input_guide_GLFW_RAW_MOUSE_MOTION}{GLFW\+\_\+\+RAW\+\_\+\+MOUSE\+\_\+\+MOTION} input mode. Raw mouse motion input is not yet implemented on mac\+OS. Call \doxylink{group__input_ga9b83d8957a566b22b3b768c7d1d4be03}{glfw\+Raw\+Mouse\+Motion\+Supported} to check if GLFW can provide raw mouse motion on the current system.

For more information see \doxysectlink{input_guide_raw_mouse_motion}{Raw mouse motion}{2}.\hypertarget{news_joysticks_33}{}\doxysubsubsection{\texorpdfstring{Joystick hats}{Joystick hats}}\label{news_joysticks_33}
GLFW can now return the state of hats (i.\+e. POVs or D-\/pads) of a joystick with \doxylink{group__input_gaf0f00897c4d010af7bd1aea1f6193e6f}{glfw\+Get\+Joystick\+Hats}. For compatibility, hats are also exposed as buttons. This can be disabled with the \doxylink{intro_guide_GLFW_JOYSTICK_HAT_BUTTONS}{GLFW\+\_\+\+JOYSTICK\+\_\+\+HAT\+\_\+\+BUTTONS} initialization hint.

For more information see \doxysectlink{input_guide_joystick_hat}{Joystick hat states}{2}.\hypertarget{news_geterror_33}{}\doxysubsubsection{\texorpdfstring{Error query}{Error query}}\label{news_geterror_33}
GLFW now supports querying the last error code for the calling thread and its human-\/readable description with \doxylink{group__init_gaee7bac4c7e1b2769207747c1b5ae2f9e}{glfw\+Get\+Error}. This can be used instead of or together with the error callback.

For more information see \doxysectlink{intro_guide_error_handling}{Error handling}{1}.\hypertarget{news_init_hints_33}{}\doxysubsubsection{\texorpdfstring{Support for initialization hints}{Support for initialization hints}}\label{news_init_hints_33}
GLFW now supports setting library initialization hints with \doxylink{group__init_gac85e41aab1c2bd6fc740f4377d9e7e9f}{glfw\+Init\+Hint}. These must be set before initialization to take effect. Some of these hints are platform specific but are safe to set on any platform.

For more information see \doxysectlink{intro_guide_init_hints}{Initialization hints}{2}.\hypertarget{news_attention_33}{}\doxysubsubsection{\texorpdfstring{User attention request}{User attention request}}\label{news_attention_33}
GLFW now supports requesting user attention with \doxylink{group__window_gad0c9629abb49447bedd28080642b8538}{glfw\+Request\+Window\+Attention}. Where possible this calls attention to the specified window. On platforms like mac\+OS it calls attention to the whole application.

For more information see \doxysectlink{window_guide_window_attention}{Window attention request}{2}.\hypertarget{news_maximize_33}{}\doxysubsubsection{\texorpdfstring{Window maximization callback}{Window maximization callback}}\label{news_maximize_33}
GLFW now supports notifying the application that the window has been maximized \doxylink{group__window_gae77b3960dcd40514fb7b2b8a6eb98137}{glfw\+Set\+Window\+Maximize\+Callback}. This is called both when the window was maximized by the user and when it was done with \doxylink{group__window_ga4f825a55367d3fabde3d06e7f30128e8}{glfw\+Maximize\+Window}.

For more information see \doxysectlink{window_guide_window_maximize}{Window maximization}{2}.\hypertarget{news_workarea_33}{}\doxysubsubsection{\texorpdfstring{Query for the monitor work area}{Query for the monitor work area}}\label{news_workarea_33}
GLFW now supports querying the work area of a monitor, i.\+e. the area not occupied by task bars or global menu bars, with \doxylink{group__monitor_ga5a3ac0f6e3786df47aabccfb5760b31d}{glfw\+Get\+Monitor\+Workarea}. On platforms that lack this concept, the whole area of the monitor is returned.

For more information see \doxysectlink{monitor_guide_monitor_workarea}{Work area}{2}.\hypertarget{news_transparency_33}{}\doxysubsubsection{\texorpdfstring{Transparent windows and framebuffers}{Transparent windows and framebuffers}}\label{news_transparency_33}
GLFW now supports the creation of windows with transparent framebuffers on systems with desktop compositing enabled with the \doxylink{group__window_ga60a0578c3b9449027d683a9c6abb9f14}{GLFW\+\_\+\+TRANSPARENT\+\_\+\+FRAMEBUFFER} window hint and attribute. This hint must be set before window creation and leaves any window decorations opaque.

GLFW now also supports whole window transparency with \doxylink{group__window_gad124d57a1c1c519f60c9765cd0c2a89c}{glfw\+Get\+Window\+Opacity} and \doxylink{group__window_ga62a784fe328f227727cbcd0d07480ff3}{glfw\+Set\+Window\+Opacity}. This value controls the opacity of the whole window including decorations and unlike framebuffer transparency can be changed at any time after window creation.

For more information see \doxysectlink{window_guide_window_transparency}{Window transparency}{2}.\hypertarget{news_key_scancode_33}{}\doxysubsubsection{\texorpdfstring{Query for the scancode of a key}{Query for the scancode of a key}}\label{news_key_scancode_33}
GLFW now supports querying the platform dependent scancode of any physical key with \doxylink{group__input_ga3ea3b9d74f8352506068620980e01fe5}{glfw\+Get\+Key\+Scancode}.

For more information see \doxysectlink{input_guide_input_key}{Key input}{2}.\hypertarget{news_center_cursor_33}{}\doxysubsubsection{\texorpdfstring{Cursor centering window hint}{Cursor centering window hint}}\label{news_center_cursor_33}
GLFW now supports controlling whether the cursor is centered over newly created full screen windows with the \doxylink{window_guide_GLFW_CENTER_CURSOR_hint}{GLFW\+\_\+\+CENTER\+\_\+\+CURSOR} window hint. It is enabled by default.\hypertarget{news_cursor_hover_33}{}\doxysubsubsection{\texorpdfstring{Mouse cursor hover window attribute}{Mouse cursor hover window attribute}}\label{news_cursor_hover_33}
GLFW now supports polling whether the cursor is hovering over the window content area with the \doxylink{window_guide_GLFW_HOVERED_attrib}{GLFW\+\_\+\+HOVERED} window attribute. This attribute corresponds to the \doxysectlink{input_guide_cursor_enter}{cursor enter/leave}{2} event.\hypertarget{news_focusonshow_33}{}\doxysubsubsection{\texorpdfstring{Window hint and attribute for input focus on show}{Window hint and attribute for input focus on show}}\label{news_focusonshow_33}
GLFW now has the \doxylink{window_guide_GLFW_DECORATED_hint}{GLFW\+\_\+\+FOCUS\+\_\+\+ON\+\_\+\+SHOW} window hint and attribute for controlling whether a window gets input focus when shown. It is enabled by default. It applies both when creating an visible window with \doxylink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window} and when showing it with \doxylink{group__window_ga7945bcdff9e5e058cf36505d6873ed8c}{glfw\+Show\+Window}.

This is a workaround for GLFW 3.\+0 lacking \doxylink{group__window_ga0da0e3daaa2d100f44a115c09077b510}{glfw\+Focus\+Window} and will be corrected in the next major version.

For more information see \doxysectlink{window_guide_window_hide}{Window visibility}{2}.\hypertarget{news_device_userptr_33}{}\doxysubsubsection{\texorpdfstring{Monitor and joystick user pointers}{Monitor and joystick user pointers}}\label{news_device_userptr_33}
GLFW now supports setting and querying user pointers for connected monitors and joysticks with \doxylink{group__monitor_ga6d89c74c1c293f0ab78a00c3c6d1ae44}{glfw\+Set\+Monitor\+User\+Pointer}, \doxylink{group__monitor_ga13cb11e8f507dbc477e10a040f0fad7f}{glfw\+Get\+Monitor\+User\+Pointer}, \doxylink{group__input_ga3a8999c2d2634c7bcf4a04f1ec64895c}{glfw\+Set\+Joystick\+User\+Pointer} and \doxylink{group__input_ga95c348e4b4308c44c59825def0d1367e}{glfw\+Get\+Joystick\+User\+Pointer}.

For more information see \doxysectlink{monitor_guide_monitor_userptr}{User pointer}{2} and \doxysectlink{input_guide_joystick_userptr}{Joystick user pointer}{2}.\hypertarget{news_macos_nib_33}{}\doxysubsubsection{\texorpdfstring{mac\+OS menu bar from nib file}{macOS menu bar from nib file}}\label{news_macos_nib_33}
GLFW will now load a {\ttfamily Main\+Menu.\+nib} file if found in the {\ttfamily Contents/\+Resources} directory of the application bundle, as a way to replace the GLFW menu bar without recompiling GLFW. This behavior can be disabled with the \doxylink{intro_guide_GLFW_COCOA_MENUBAR_hint}{GLFW\+\_\+\+COCOA\+\_\+\+MENUBAR} initialization hint.\hypertarget{news_glext_33}{}\doxysubsubsection{\texorpdfstring{Support for more context creation extensions}{Support for more context creation extensions}}\label{news_glext_33}
The context hint \doxylink{window_guide_GLFW_SRGB_CAPABLE}{GLFW\+\_\+\+SRGB\+\_\+\+CAPABLE} now supports Open\+GL ES via {\ttfamily WGL\+\_\+\+EXT\+\_\+colorspace}, the context hint \doxylink{group__window_ga5a52fdfd46d8249c211f923675728082}{GLFW\+\_\+\+CONTEXT\+\_\+\+NO\+\_\+\+ERROR} now supports {\ttfamily WGL\+\_\+\+ARB\+\_\+create\+\_\+context\+\_\+no\+\_\+error} and {\ttfamily GLX\+\_\+\+ARB\+\_\+create\+\_\+context\+\_\+no\+\_\+error}, the context hint \doxylink{group__window_ga72b648a8378fe3310c7c7bbecc0f7be6}{GLFW\+\_\+\+CONTEXT\+\_\+\+RELEASE\+\_\+\+BEHAVIOR} now supports {\ttfamily EGL\+\_\+\+KHR\+\_\+context\+\_\+flush\+\_\+control} and \doxylink{group__context_ga0e8af175218929615c16e74938c10f2a}{glfw\+Get\+Proc\+Address} now supports {\ttfamily EGL\+\_\+\+KHR\+\_\+get\+\_\+all\+\_\+proc\+\_\+addresses}.\hypertarget{news_osmesa_33}{}\doxysubsubsection{\texorpdfstring{OSMesa off-\/screen context creation support}{OSMesa off-screen context creation support}}\label{news_osmesa_33}
GLFW now supports creating off-\/screen Open\+GL contexts using \href{https://www.mesa3d.org/osmesa.html}{\texttt{ OSMesa}} by setting \doxylink{window_guide_GLFW_CONTEXT_CREATION_API_hint}{GLFW\+\_\+\+CONTEXT\+\_\+\+CREATION\+\_\+\+API} to {\ttfamily GLFW\+\_\+\+OSMESA\+\_\+\+CONTEXT\+\_\+\+API}. Native access function have been added to retrieve the OSMesa color and depth buffers.

There is also a new null backend that uses OSMesa as its native context creation API, intended for automated testing. This backend does not provide input.\hypertarget{news_caveats_33}{}\doxysubsection{\texorpdfstring{Caveats for version 3.\+3}{Caveats for version 3.3}}\label{news_caveats_33}
\hypertarget{news_joystick_layout_33}{}\doxysubsubsection{\texorpdfstring{Layout of joysticks have changed}{Layout of joysticks have changed}}\label{news_joystick_layout_33}
The way joystick elements are arranged have changed to match SDL2 in order to support SDL\+\_\+\+Game\+Controller\+DB mappings. The layout of joysticks may change again if required for compatibility with SDL2. If you need a known and stable layout for game controllers, see if you can switch to \doxysectlink{input_guide_gamepad}{Gamepad input}{2}.

Existing code that depends on a specific joystick layout will likely have to be updated.\hypertarget{news_wait_events_33}{}\doxysubsubsection{\texorpdfstring{No window required to wait for events}{No window required to wait for events}}\label{news_wait_events_33}
The \doxylink{group__window_ga6e042d05823c11e11c7339b81a237738}{glfw\+Wait\+Events} and \doxylink{group__window_ga05223a0a4c5e50f10f289e60398153aa}{glfw\+Wait\+Events\+Timeout} functions no longer need a window to be created to wait for events. Before version 3.\+3 these functions would return immediately if there were no user-\/created windows. On platforms where only windows can receive events, an internal helper window is used.

Existing code that depends on the earlier behavior will likely have to be updated.\hypertarget{news_gamma_ramp_size_33}{}\doxysubsubsection{\texorpdfstring{Gamma ramp size of 256 may be rejected}{Gamma ramp size of 256 may be rejected}}\label{news_gamma_ramp_size_33}
The documentation for versions before 3.\+3 stated that a gamma ramp size of 256 would always be accepted. This was never the case on X11 and could lead to artifacts on mac\+OS. The \doxylink{group__monitor_ga3e4ab484476c935b4cd0bf49a5c429d6}{glfw\+Set\+Gamma} function has been updated to always generate a ramp of the correct size.

Existing code that hardcodes a size of 256 should be updated to use the size of the current ramp of a monitor when setting a new ramp for that monitor.\hypertarget{news_xinput_deadzone_33}{}\doxysubsubsection{\texorpdfstring{Windows XInput deadzone removed}{Windows XInput deadzone removed}}\label{news_xinput_deadzone_33}
GLFW no longer applies any deadzone to the input state received from the XInput API. This was never done for any other platform joystick API so this change makes the behavior more consistent but you will need to apply your own deadzone if desired.\hypertarget{news_x11_clipboard_33}{}\doxysubsubsection{\texorpdfstring{X11 clipboard transfer limits}{X11 clipboard transfer limits}}\label{news_x11_clipboard_33}
GLFW now supports reading clipboard text via the {\ttfamily INCR} method, which removes the limit on how much text can be read with \doxylink{group__input_ga315b28b05797d00fe7cdf1ecfdc638dc}{glfw\+Get\+Clipboard\+String}. However, writing via this method is not yet supported, so you may not be able to write a very large string with \doxylink{group__input_ga7a580309bbc185a0459c3559021d2fd7}{glfw\+Set\+Clipboard\+String} even if you read it from the clipboard earlier.

The exact size limit for writing to the clipboard is negotiated with each receiving application but is at least several tens of kilobytes. Note that only the read limit has changed. Any string that could be written before still can be.\hypertarget{news_x11_linking_33}{}\doxysubsubsection{\texorpdfstring{X11 extension libraries are loaded dynamically}{X11 extension libraries are loaded dynamically}}\label{news_x11_linking_33}
GLFW now loads all X11 extension libraries at initialization. The only X11 library you need to link against is {\ttfamily lib\+X11}. The header files for the extension libraries are still required for compilation.

Existing projects and makefiles that link GLFW directly against the extension libraries should still build correctly but will add these libraries as load-\/time dependencies.\hypertarget{news_cmake_version_33}{}\doxysubsubsection{\texorpdfstring{CMake 3.\+0 or later is required}{CMake 3.0 or later is required}}\label{news_cmake_version_33}
The minimum CMake version has been raised from 2.\+8.\+12 to 3.\+0. This is only a requirement of the GLFW CMake files. The GLFW source files do not depend on CMake.\hypertarget{news_caveat_fbtransparency_33}{}\doxysubsubsection{\texorpdfstring{Framebuffer transparency requires DWM transparency}{Framebuffer transparency requires DWM transparency}}\label{news_caveat_fbtransparency_33}
GLFW no longer supports framebuffer transparency enabled via \doxylink{group__window_ga60a0578c3b9449027d683a9c6abb9f14}{GLFW\+\_\+\+TRANSPARENT\+\_\+\+FRAMEBUFFER} on Windows 7 if DWM transparency is off (the Transparency setting under Personalization \texorpdfstring{$>$}{>} Window Color).\hypertarget{news_emptyevents_33}{}\doxysubsubsection{\texorpdfstring{Empty events on X11 no longer roundtrip to server}{Empty events on X11 no longer roundtrip to server}}\label{news_emptyevents_33}
Starting with GLFW 3.\+3.\+7, events posted with \doxylink{group__window_gad6f6b997278c91c43b35970eb9713d7e}{glfw\+Post\+Empty\+Event} now use a separate unnamed pipe instead of sending an X11 client event to the helper window.\hypertarget{news_deprecations_33}{}\doxysubsection{\texorpdfstring{Deprecations in version 3.\+3}{Deprecations in version 3.3}}\label{news_deprecations_33}
\hypertarget{news_charmods_callback_33}{}\doxysubsubsection{\texorpdfstring{Character with modifiers callback}{Character with modifiers callback}}\label{news_charmods_callback_33}
The character with modifiers callback set with \doxylink{group__input_ga359a8529c2eb06e6e2173168f30ac99c}{glfw\+Set\+Char\+Mods\+Callback} has been deprecated and should if possible not be used.

Existing code should still work but further bug fixes will likely not be made. The callback will be removed in the next major version.\hypertarget{news_clipboard_window_33}{}\doxysubsubsection{\texorpdfstring{Window parameter to clipboard functions}{Window parameter to clipboard functions}}\label{news_clipboard_window_33}
The window parameter of the clipboard functions \doxylink{group__input_ga315b28b05797d00fe7cdf1ecfdc638dc}{glfw\+Get\+Clipboard\+String} and \doxylink{group__input_ga7a580309bbc185a0459c3559021d2fd7}{glfw\+Set\+Clipboard\+String} has been deprecated and is no longer used on any platform. On platforms where the clipboard must be owned by a specific window, an internal helper window is used.

Existing code should still work unless it depends on a specific window owning the clipboard. New code may pass {\ttfamily NULL} as the window argument. The parameter will be removed in a future release.\hypertarget{news_removals_33}{}\doxysubsection{\texorpdfstring{Removals in 3.\+3}{Removals in 3.3}}\label{news_removals_33}
\hypertarget{news_macos_options_33}{}\doxysubsubsection{\texorpdfstring{mac\+OS specific CMake options and macros}{macOS specific CMake options and macros}}\label{news_macos_options_33}
The {\ttfamily GLFW\+\_\+\+USE\+\_\+\+RETINA}, {\ttfamily GLFW\+\_\+\+USE\+\_\+\+CHDIR} and {\ttfamily GLFW\+\_\+\+USE\+\_\+\+MENUBAR} CMake options and the {\ttfamily \+\_\+\+GLFW\+\_\+\+USE\+\_\+\+RETINA}, {\ttfamily \+\_\+\+GLFW\+\_\+\+USE\+\_\+\+CHDIR} and {\ttfamily \+\_\+\+GLFW\+\_\+\+USE\+\_\+\+MENUBAR} compile-\/time macros have been removed.

These options and macros are replaced by the window hint \doxylink{window_guide_GLFW_COCOA_RETINA_FRAMEBUFFER_hint}{GLFW\+\_\+\+COCOA\+\_\+\+RETINA\+\_\+\+FRAMEBUFFER} and the init hints \doxylink{intro_guide_GLFW_COCOA_CHDIR_RESOURCES_hint}{GLFW\+\_\+\+COCOA\+\_\+\+CHDIR\+\_\+\+RESOURCES} and \doxylink{intro_guide_GLFW_COCOA_MENUBAR_hint}{GLFW\+\_\+\+COCOA\+\_\+\+MENUBAR}.

Existing projects and makefiles that set these options or define these macros during compilation of GLFW will still build but it will have no effect and the default behaviors will be used.\hypertarget{news_vulkan_sdk_33}{}\doxysubsubsection{\texorpdfstring{Lunar\+G Vulkan SDK dependency}{LunarG Vulkan SDK dependency}}\label{news_vulkan_sdk_33}
The GLFW test programs that previously depended on the LunarG Vulkan SDK now instead uses a Vulkan loader generated by \href{https://github.com/Dav1dde/glad}{\texttt{ glad2}}. This means the GLFW CMake files no longer look for the Vulkan SDK.

Existing CMake projects that depended on the Vulkan SDK cache variables from GLFW will need to call {\ttfamily find\+\_\+package(\+Vulkan)} themselves. CMake 3.\+7 and later already comes with a \href{https://cmake.org/cmake/help/latest/module/FindVulkan.html}{\texttt{ Vulkan find module}} similar to the one GLFW previously included.\hypertarget{news_lib_suffix_33}{}\doxysubsubsection{\texorpdfstring{CMake option LIB\+\_\+\+SUFFIX}{CMake option LIB\_SUFFIX}}\label{news_lib_suffix_33}
The {\ttfamily LIB\+\_\+\+SUFFIX} CMake option has been removed. GLFW now uses the GNUInstall\+Dirs CMake package to handle platform specific details like the library directory suffix and the {\ttfamily LIB\+\_\+\+SUFFIX} CMake option has been removed.

Existing projects and makefiles that set the {\ttfamily LIB\+\_\+\+SUFFIX} option will use the suffix chosen by the GNUInstall\+Dirs package and the option will be ignored.\hypertarget{news_wl_shell_33}{}\doxysubsubsection{\texorpdfstring{Support for the wl\+\_\+shell protocol}{Support for the wl\_shell protocol}}\label{news_wl_shell_33}
